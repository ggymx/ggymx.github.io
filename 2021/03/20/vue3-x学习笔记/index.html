废弃filter

废弃原因：
虽然这看起来很方便，但它需要一个自定义语法，打破大括号内表达式是“只是 JavaScript”的假设，这不仅有学习成本，而且有实现成本

完全废除filter，替换方案是用计算属性或函数，全局filter定义可以替换为全局prop定义

增加片段概念

增加了片段的概念，即组件支持返回多个节点

对片段类型组件的attr绑定可以使用v-bind=”$attrs”

template组件的性能提升

vue2.x中的函数组件的渲染速度比template组件要快，因此无状态的简单组件可以使用函数组件
而3.x的template的组件的性能优化已经和函数组件相差无几，因此函数组件的优势仅仅只剩定义简单组件

无法再默认实现eventBus

$on，$off 和 $once 实例方法已被移除，应用实例不再实现事件触发接口。
即vue实例已经废弃了这三个方法，即通过创建 event hub，以创建在整个应用程序中使用的全局事件侦听器已经行不通了，即3.x无法再利用vue的这些特性去创建一个enevtBus(事件总线)>
需要使用eventBus可以显式引入实现事件发射器接口的外部库：如mitt

自定义指令生命周期方法命名更新

将自定义指令的生命周期方法的命名更新和组件生命周期方法近似一致，减少记忆成本

全局配置迁移

Vue全局配置的全局API迁移

3.x中全局配置的API由Vue迁移到createApp返回的vue实例上

迁移理由：
2.x中Vue的全局配置会影响所有的vue实例，会强制污染一些不想有此配置的vue实例，如测试时定义的vue实例也会受到此全局配置的影响，因此对测试也带来了麻烦

全局API tree shaking

3.x中实现了非全局配置的全局API按需引入，有利于编译器编译时tree-shaking（性能优化）

Vue.nextTick() 是一个全局的 API 直接暴露在单个 Vue 对象上——事实上，实例方法 $nextTick() 只是一个方便的包装 Vue.nextTick() 为方便起见，回调的 this 上下文自动绑定到当前实例。

v-if和v-for优先级更改

优先级：
v2.x：v-for>v-if
v3.x：v-if>v-for

由于语法上存在歧义，建议避免在同一元素上同时使用两者。

比起在模板层面管理相关逻辑，更好的办法是通过创建计算属性筛选出列表，并以此创建可见元素。

更改v-bind合并行为

不兼容：v-bind 的绑定顺序会影响渲染结果。

在元素上动态绑定 attribute 时，常见的场景是在一个元素中同时使用 v-bind=”object” 语法和单独的 property。然而，这就引出了关于合并的优先级的问题。
在 2.x，如果一个元素同时定义了 v-bind=”object” 和一个相同的单独的 property，那么这个单独的 property 总是会覆盖 object 中的绑定。

在 3.x，如果一个元素同时定义了 v-bind=”object” 和一个相同的单独的 property，那么声明绑定的顺序决定了它们如何合并（开发者对自己所希望的合并行为有了更好的控制。）
