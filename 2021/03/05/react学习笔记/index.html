最终编译

React 认为渲染逻辑本质上与其他 UI 逻辑内在耦合，比如，在 UI 中需要绑定处理事件、在某些时刻状态发生变化时需要通知到 UI，以及需要在 UI 中展示准备好的数据。
React 并没有采用将标记与逻辑进行分离到不同文件这种人为地分离方式，而是通过将二者共同存放在称之为“组件”的松散耦合单元之中，来实现关注点分离。
在 JSX 语法中，你可以在大括号内放置任何有效的 JavaScript 表达式

JSX 特定属性
在属性中嵌入 JavaScript 表达式时，不要在大括号外面加上引号。你应该仅使用引号（对于字符串值）或大括号（对于表达式）中的一个，对于同一属性不能同时使用这两种符号。
JSX 遵循小驼峰命名（camelCase），如 class 样式写成 className tabindex 写成 tabIndex

元素渲染

元素是构成 React 应用的最小砖块。
与浏览器的 DOM 元素不同，React 元素是创建开销极小的普通对象。React DOM 会负责更新 DOM 来与 React 元素保持一致
使用 React 构建的应用通常只有单一的根 DOM 节点。
react 元素是不可变对象。一旦被创建，你就无法更改它的子元素或者属性。一个元素就像电影的单帧：它代表了某个特定时刻的 UI。
改变 react 元素后，可以使用 ReactDOM.render()更新 UI 但大多数 React 应用只会调用一次 ReactDOM.render()，因此需要设置组件的状态来更新 UI

注：
diff 算法：React DOM 会将元素和它的子元素与它们之前的状态进行比较，并只会进行必要的更新来使 DOM 达到预期的状态。

React.PureComponent

React.PureComponent 与 React.Component 几乎完全相同，但 React.PureComponent 通过props和state的浅对比来实现 shouldComponentUpate()。如果对象包含复杂的数据结构，它可能会因深层的数据不一致而产生错误的否定判断(表现为对象深层的数据已改变视图却没有更新）

关注点：

  1. 无论组件是否是 PureComponent，如果定义了 shouldComponentUpdate()，那么会调用它并以它的执行结果来判断是否 update。在组件未定义 shouldComponentUpdate() 的情况下，会判断该组件是否是 PureComponent，如果是的话，会对新旧 props、state 进行 shallowEqual 比较，一旦新旧不一致，会触发 update。
  2. 浅比较 只会比较到两个对象的 ownProperty 是否符合 Object.js() 判等，不会递归地去深层比较
  3. 至于复杂数据结构，用Object.key()获取下key，然后key和对应的value都是基础类型数据，就是算是简单数据结构，不然就是复杂

数据是向下流动的

这通常会被叫做“自上而下”或是“单向”的数据流。任何的 state 总是所属于特定的组件，而且从该 state 派生的任何数据或 UI 只能影响树中“低于”它们的组件。

总结：

  • class 组件应该始终使用 props 参数来调用父类的构造函数。
  • 挂载在组件的功能，要在组件卸载时及时卸载（防止内存泄漏）
  • 不要直接修改 state，state 只能在 class 的构造函数中被赋初值，其他地方直接修改 state 不会有响应
  • 更新状态需要使用 setState 方法，setState 会触发组件的 render 方法，从而引起 UI 重绘
  • 组件的 state、props 的更新可能是异步的，不要依赖他们的值来更新下一个状态。
  • 可以使用 setState 单独更新要更新的属性，state 的更新会被合并
  • 数据是向下流动的（单向数据流），从该 state 派生的任何数据或 UI 只能影响树中“低于”它们的组件。
  • 任何的 state 总是所属于特定的组件，具有封装性和局部性
  • 可以向 class 组件中随意添加不参与数据流（比如计时器 ID）的额外字段。

注意事项

不要在 render 方法中使用 HOC
React 的 diff 算法（称为协调）使用组件标识来确定它是应该更新现有子树还是将其丢弃并挂载新子树。 如果从 render 返回的组件与前一个渲染中的组件相同（===），则 React 通过将子树与新子树进行区分来递归更新子树。 如果它们不相等，则完全卸载前一个子树。

通常，你不需要考虑这点。但对 HOC 来说这一点很重要，因为这代表着你不应在组件的 render 方法中对一个组件应用 HOC：

这不仅仅是性能问题 - 重新挂载组件会导致该组件及其所有子组件的状态丢失。
如果在组件之外创建 HOC，这样一来组件只会创建一次。因此，每次 render 时都会是同一个组件。一般来说，这跟你的预期表现是一致的。
在极少数情况下，你需要动态调用 HOC。你可以在组件的生命周期方法或其构造函数中进行调用。

React.PureComponent 一起使用时要小心

PureComponent自带通过props和state的浅对比来实现 shouldComponentUpate()，而Component没有。

PureComponent缺点
可能会因深层的数据不一致而产生错误的否定判断，从而shouldComponentUpdate结果返回false，界面得不到更新。

PureComponent优势
不需要开发者自己实现shouldComponentUpdate，就可以进行简单的判断来提升性能。

你在 render 方法里创建函数，那么使用 render prop 会抵消使用 React.PureComponent 带来的优势。因为浅比较 props 的时候总会得到 false，并且在这种情况下每一个 render 对于 render prop 将会生成一个新的值。

hook设计的基石：只要在在组件顶层的hook依次执行，就能使state能正确关联（解释了为什么hook不能在if，循环以及非自定义hook函数体内）

注意

  • hook只能用于函数组件中，且必须在函数组件顶层或者自定义hook中使用，不能出现在条件，循环以及其他函数中

  • 代码复用：自定义hook，HOC（高阶组件），render props是代码复用的途径（类似于minxins，react中已经逐渐抛弃minxins）

  • hook内对组件的state会形成一个闭包，因此如果没有及时执行hook的话，该state不会更新，可以在hook传入的函数中添加参数

  • hooks可以多次添加，会依次执行，useEffect等副作用hook会在组件渲染完成后执行，其他hook在渲染期间执行

  • react有计划默认根据hook的传入函数的参数自动生成依赖项，而不用手动配置，这样会更智能

  • 可以添加相应的eslint插件来添加hooks运行时的错误提示以及修改建议

副作用hook

副作用函数，代表处理组件中的副作用，副作用的概念是针对于纯组件而言
常见场景：对dom操作，数据获取，添加计时器，添加订阅，其中计时器和订阅需要显式卸载

useEffect

相当于componentDidMount componentDidUpdate componentWillUnmount的总和，在组件渲染完成之后异步调用，可以添加依赖项，来指定当依赖项没有变化时跳过更新（性能优化），如果返回一个函数，则会在卸载时调用该函数

useEffect内传入的函数所用的参数，都要作为依赖项, useEffect((a,b)=>{},[a,b])，这常是bug的来源

仅在组件首次渲染和卸载时有效：useEffect(()=>{},[])，这不依赖任何props和state，因此该effect仅在组件首次渲染和卸载时有效

useLayoutEffect

useLayoutEffect相当于componentDidMount componentDidUpdate，会在组件渲染完成后同步调用，当遇到需要立刻更新的UI操作时能用到它

思路革新：useEffect，useLayoutEffect的产生可以将相关的副作用代码关联在一起，而不用拆分到不同的生命周期方法中，使代码逻辑更加紧凑，也改变了严格按照生命周期划分组件状态的思路

逻辑封装复用

在 React 中有两种流行的方式来共享组件之间的状态逻辑: render props 和高阶组件，现在让我们来看看Hook是如何在让你不增加组件的情况下解决相同问题的。

当我们想在两个函数之间共享逻辑时，我们会把它提取到第三个函数中。而组件和 Hook 都是函数，所以也同样适用这种方式。

自定义 Hook 是一个函数，其名称以 “use” 开头，函数内部可以调用其他的 Hook。
