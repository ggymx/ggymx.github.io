uniapp之平台差异梳理

2020-08-16
uniapp, vue

uniapp：一款优秀的跨平台开发框架

创建项目方式

  • HubilderX创建，编译器在HubilderX中
  • cli创建，编译器在项目中
    ps:两种编译器都有的话，默认使用项目中的编译器

插件安装方式

  • NPM安装（兼容性不好，往往只支持h5端，npm方式不支持小程序自定义组件）
  • 插件市场安装（兼容性好）

开发规范

为了实现多端兼容，综合考虑编译速度、运行性能等因素，uni-app 约定了如下开发规范：

页面文件遵循 Vue 单文件组件 (SFC) 规范
组件标签靠近小程序规范，详见uni-app 组件规范
接口能力（JS API）靠近微信小程序规范，但需将前缀 wx 替换为 uni，详见uni-app接口规范
数据绑定及事件处理同 Vue.js 规范，同时补充了App及页面的生命周期
为兼容多端运行，建议使用flex布局进行开发

运行环境判断

  • 在HBuilderX 中，点击“运行”编译出来的代码是开发环境，点击“发行”编译出来的代码是生产环境
  • cli模式下，是通行的编译环境处理方式。

ps:process.env.NODE_ENV development：开发环境，production：生产环境
可以自定义更多环境，比如测试环境

运行期判断：运行期判断是指代码已经打入包中，仍然需要在运行期判断平台，此时可使用 uni.getSystemInfoSync().platform 判断客户端环境是 Android、iOS 还是小程序开发工具（在百度小程序开发工具、微信小程序开发工具、支付宝小程序开发工具中使用 uni.getSystemInfoSync().platform 返回值均为 devtools）。

如有必要，也可以在条件编译里自己定义一个变量，赋不同值。在后续运行代码中动态判断环境

尺寸单位支持

  • uniapp的vue页面支持通用css单位，还有upx（uniapp早期的自适应动态单位，不建议使用），rpx（微信的自适应动态单位，uniapp官方推荐）
    一个页面屏幕宽度定义为750rpx/750upx
    此外还有vh（视窗高度，1vh等于视窗高度的1%），vw（视窗宽度，1vw等于视窗宽度的1%）

  • nvue在uniapp模式下（支持原生渲染和webview渲染）可以使用rpx，px单位，但自weex模式下（weex模式下只支持原生渲染），weex只支持px（相当于rpx）和wx（相当于px）

uniapp渲染方式

webview渲染：vue页面（web前端，不熟悉原生排版，建议以此为主，在App端某些vue页面表现不佳的场景下使用 nvue 作为强化补充）
原生渲染：nvue页面（weex或react native 开发者建议使用）
原生渲染针对app端，app端默认不开启纯原生渲染，即app中可以使用nvue和vue共同开发
manifest.json源码视图的”app-plus”下配置”renderer”:”native”即开启纯原生渲染模式，只支持nvue开发（此时pages.json注册的vue页面将被忽略，vue组件也将被原生渲染引擎来渲染。）
nvue适用于：高性能的特定区域长列表（如聊天记录·），软键盘按钮文字（如聊天工具）复杂高性能的自定义下拉刷新，左右拖动长列表，
解决前端控件无法覆盖原生控件的层级问题，深度使用map组件，深度使用video，直播推流，对app启动速度要求极致化
缺点：canvas性能不高，尤其是Android App平台，所以这个组件干脆没有内置，而是需要单独引入。（操作canvas动画，最高性能的方式是使用vue页面的renderjs技术）
动态横竖屏：nvue页面的css不支持媒体查询，所以横竖屏动态切换、动态适配屏幕很困难

uniapp编译模式

weex编译模式：仅适用于app（默认是weex？）
uni-app编译模式：所有端，包含小程序和H5
在 manifest.json 中修改2种编译模式，manifest.json -> app-plus -> nvueCompiler 切换编译模式。

uniapp的自定义组件处理方式

自定义组件模式：即打包时在小程序端将SFC（vue单文件组件）打包为相应的自定义组件
非自定义组件模式：即打包时在小程序端将SFC（vue单文件组件）打包为相应的template模板（参照了mpvue，在打包时将组件编译为小程序模板，原因是小程序早期不支持自定义组件模式），老版本的uniapp默认这种方式，但小程序支持自定义组件
后，uniapp渐渐废弃了这种方式，现在只支持自定义组件模式（但此模式已于2019年11月起下线。）
manifest中"usingComponents":false

ES6支持

uni-app 在支持绝大部分 ES6 API 的同时，也支持了 ES7 的 await/async。
因为iOS上不允许三方js引擎，所以iOS上不区分App、小程序、H5，各端均仅依赖iOS版本。
各端Android版本有差异
Array的includes ios8不支持，values ios8和安卓不支持 String的normalize在ios10以下不支持，proxy安卓和ios10以下都不支持

在 vue 文件的 script 节点声明 lang=”ts”
声明 lang=”ts” 后，该 vue 文件 import 进来的所有 vue 组件，均需要使用 ts 编写

其他

模板语法

uniapp高度支持vue模板语法，在老版的非自定义组件模式下部分模版语法不支持
生命周期：uniapp支持vue组件实例的生命周期（vue应用概念）和应用/页面生命周期（小程序概念）

页面跳转

uniapp支持小程序类跳转或vue-router（需要自行配置）

全局状态管理

1.vuex
vuex 是一个专为 Vue.js 应用程序开发的状态管理模式。它采用集中式存储管理应用的所有组件的状态，并以相应的规则保证状态以一种可预测的方式发生变化。

HBuilderX 2.2.5+起，支持vue和nvue之间共享
uniapp项目默认内置vuex，使用Vue.use注册使用即可

2.公用模块
定义一个专用的模块，用来组织和管理这些全局的变量，在需要的页面引入。
示例：在 uni-app 项目根目录下创建 common 目录，然后在 common 目录下新建 helper.js 用于定义公用的方法。

注意这种方式只支持多个vue页面或多个nvue页面之间公用，vue和nvue之间不公用。

3.globalData
小程序中有个globalData概念，可以在 App 上声明全局变量。 Vue 之前是没有这类概念的，但 uni-app 引入了globalData概念，并且在包括H5、App等平台都实现了。
在 App.vue 可以定义 globalData ，也可以使用 API 读写这个值。

globalData支持vue和nvue共享数据。

globalData是一种比较简单的全局变量使用方式。

vue特性支持

涉及到dom方面的特性，h5端全部支持，app旧版支持不高，app的v3模式增加更多支持，小程序几乎不支持（webview中没有dom概念，况且小程序端逻辑层和视图层分离，也不允许直接操作dom）
涉及到data方面的特性，几乎都支持

视图层的脚本语言支持

微信小程序端提供了wxs，百度小程序提供了Filter、阿里小程序提供了SJS，uni-app也支持使用这些功能，并将它们编译到百度和阿里的小程序端。不过它们的功能还不如wxs强大。此外头条系小程序自身不支持类似功能。
它的特点是运行在视图层的脚本。可避免逻辑层和渲染层交互通信折损，大大提示渲染性能，常用于数据的格式化以及视图层功能函数定义（类似于vue的filter以及util）

UI组件

uniapp自己封装了一些常用的组件，兼容性好，在开发小程序时，如无特殊必要可以不用自定义小程序组件或者引用小程序组件，引用小程序组件时可以使用条件编译专门适配小程序
此外还有一些别的ui组件库可以按需引入（如uView，使用了easycom模式）

easycom

HBuilderX 2.5.5起支持easycom组件模式。
统vue组件，需要安装、引用、注册，三个步骤后才能使用组件。easycom将其精简为一步。 只要组件安装在项目的components目录下，并符合components/组件名称/组件名称.vue目录结构。就可以不用引用、注册，直接在页面中使用。
不管components目录下安装了多少组件，easycom打包后会自动剔除没有使用的组件，对组件库的使用尤为友好。
组件库批量安装，随意使用，自动按需打包。以官方的uni-ui为例，在HBuilderX新建项目界面选择uni-ui项目模板，只需在页面中敲u，拉出大量组件代码块，直接选择，即可使用。大幅提升开发效率，降低使用门槛。

在uni-app插件市场下载符合components/组件名称/组件名称.vue目录结构的组件，均可直接使用。

easycom是自动开启的，不需要手动开启，有需求时可以在pages.json的easycom节点进行个性化设置，如关闭自动扫描，或自定义扫描匹配组件的策略。
